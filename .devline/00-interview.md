## 六組訪談問題

### A. 目標與動機（已確立）
- **一句話**：把現行「好室開發」包租代管系統（靜態HTML入口 + SharePoint Excel當DB + n8n表單）重寫成一套自建的前後端系統，資料收進 PostgreSQL，分「內部員工」與「對外房客/房東」兩種模式。
- **為什麼現在**：現況資料散在 SharePoint Excel、無真正可篩選/匯出的資料頁、無分權、維護仰賴 n8n 表單；要一個能長期維護、資料集中、畫面可由 DB 維護的系統。

### B. 成功樣貌（可檢查）
- **首個里程碑（DMO）**：一個**通用資料檢視頁**，能「選任一張表 → 依欄位篩選 → 把篩選結果匯出成 .xlsx」，畫面無 AI 感、有美感。★使用者明示要「全做」＝一頁涵蓋所有表（元數據驅動），非每表寫死。
- **完整版（v1 內部側）**：內部員工可查詢/匯出被授權的資料；管理者可維護主檔（案場/電號/房號）與設定、上傳度數/電價、跑電費計算與繳租確認——足以一次切換取代 n8n/SharePoint。
- **第一個使用者怎麼用**：內部員工登入 → 通用檢視頁選「繳租確認明細」→ 篩選某案場/某帳單年月 → 匯出 xlsx 交會計。

### C. 範圍邊界
- **不做清單**：
  1. v1 不做對外房客/房東登入（需求保留，列 v2；現況無逐房客身分基礎）。
  2. 不做 n8n 聊天機器人（除非後續明確要）。
  3. v1 不做行動 App，只做響應式網頁。
  4. 不自造「整頁版面塞 DB」的低代碼框架（第2層），只做第1層元數據驅動（欄位/篩選/匯出設定）。
- **最小可用版（v1 分三階段交付）**：① 匯入快照 + JWT 內部雙角色 + 通用檢視/篩選/匯出頁（＝DMO）→ ② 取代 n8n 的主檔CRUD/度數電價上傳/電費計算/繳租確認 → ③ 一次切換上線。

### D. 限制（本輪已收斂）
- **預算／部署**：付費雲 PaaS（render/Zeabur/Fly 之類）；注意免費額度的休眠/DB到期限制（沿用反詐平台經驗，到期日記 log 供提醒）。
- **時程**：未明示（soft 未決，不阻擋規格）；先以「DMO 優先、盡快可看手感」為節奏。
- **技術偏好/禁區**：後端定 **FastAPI**（型別積木實戰驗證、DB 同事接 Python 較容易）；前端 Next.js + TanStack Table + SheetJS；DB PostgreSQL。

### E. 既有資產（已盤點）
- 既有線上系統 `haoshi.nexonnect.com`（admin/admin，可登入參考現行行為）。
- 4 份來源資料：房客資料表 / 歷程房客資料表 / 管理提醒專區 / 繳租表單紀錄檔(核心11表)；O365 操作手冊 PDF（業務流程+電費公式）。
- 資料模型分析報告：`scratchpad/data-analysis.md`（13表、關聯已用實際資料交集驗證、14項資料品質地雷）。

### F. 主觀品質（已確立方向，細節站1交 taste-distiller）
- 不要 AI 感：避開紫藍漸層、圓角發光、emoji 滿版。
- 走乾淨資訊型設計：沿用既有米色底(#f6f4f0)+白卡+細邊框、思源黑體、低彩度單一主色、資料表以可讀性優先。

## 訪談結論
